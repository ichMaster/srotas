# v0.1 — GitHub Issues

Issues for phase **v0.1 — Memory package + initial model.yaml** (version
**v0 — Prototype**), derived from the per-phase Tasks in
[ROADMAP.md](../../ROADMAP.md) (§0.1) and the contracts in
[ARCHITECTURE.md](../../ARCHITECTURE.md) (§Memory package, §Contracts that must
not drift). This file is scoped to a single phase; IDs start the project
(**SROTAS-001…004**).

This phase lays the foundation: the interest model becomes a real file on disk
(`memory/model.yaml`, 14 nodes from the reviewed candidates) with a code seam to
read/write it non-destructively (ruamel round-trip), and the event journal
(`memory/events.sqlite`) that makes every later model change auditable — seeded
with a `bootstrap` event recording where the model came from. Nothing here
scores, collects, or embeds; it is the memory package everything else leans on.

## v0.1 — Memory package + initial model.yaml

### SROTAS-001 — `memory/model.yaml` from the 14 reviewed candidates

**Description:**
Create the memory package's model file. Transform the 14 reviewed nodes in
[spec/model-candidates.yaml](../../model-candidates.yaml) into
`memory/model.yaml`, dropping the `evidence` and `work` metadata fields so each
node carries only the prototype node format. Touches only the `memory/`
directory (data, not code).

**What needs to be done:**
- Create `memory/model.yaml` with all 14 nodes as a top-level YAML list.
- Each node has exactly `id`, `label` (Ukrainian display string), `keywords`
  (English list), `weight` (0.05–1.0) — no `evidence`, no `work`, no other
  fields.
- Preserve a short human header comment (provenance: Lumi facts, snapshot
  2026-07-04) so the file reads as owner-editable.
- Confirm `memory/model.yaml` is git-tracked (it is *not* in the gitignore set;
  only `events.sqlite` is).

**Dependencies:** None

**Expected result:**
`memory/model.yaml` holds 14 nodes in the flat prototype format, ready to load.

**Acceptance criteria:**
- [ ] `memory/model.yaml` parses as a list of 14 nodes.
- [ ] Every node has exactly the keys `id`, `label`, `keywords`, `weight`;
      no `evidence`/`work`/other keys anywhere.
- [ ] All weights are within `[0.05, 1.0]`; all ids are unique.
- [ ] **Unit test:** a data-shape test loads the checked-in `memory/model.yaml`
      and asserts 14 nodes, the exact key set, unique ids, and weight bounds
      (no paid/network call).
- [ ] Ties to the phase DoD ("`model.yaml` loads into 14 Node objects").

### SROTAS-002 — `core/model.py` — Node + ruamel round-trip read/write

**Description:**
The model reader/writer seam. `core/model.py` defines the `Node` class and loads
`memory/model.yaml` into `Node` objects; a write path updates a node's `weight`
via **ruamel.yaml** in round-trip mode so the human's comments and formatting
survive a code write. This pins the **flat node-format contract** (§Contracts:
`id`, `label`, `keywords`, `weight`, nothing else).

**What needs to be done:**
- `Node` dataclass/class with fields `id`, `label`, `keywords`, `weight` — no
  kind/tier/half_life/related, no graph, no decay.
- `load_model(path) -> list[Node]` reading `memory/model.yaml`.
- `set_weight(path, node_id, weight)` (or equivalent) that writes back the
  `weight` scalar only, via ruamel round-trip, leaving comments/formatting
  intact. No pyyaml.
- Reject/ignore unknown node fields per the flat contract (fail loudly if the
  file grows a field the prototype forbids).

**Dependencies:** SROTAS-001

**Expected result:**
`load_model("memory/model.yaml")` returns 14 `Node` objects; a weight write
round-trips the file without disturbing comments.

**Acceptance criteria:**
- [ ] `load_model` returns 14 `Node`s from the checked-in `memory/model.yaml`.
- [ ] **Contract test:** the flat node format is pinned — a `Node` exposes
      exactly `id`, `label`, `keywords`, `weight`; a fixture with an extra field
      (e.g. `tier`) is rejected. (ARCHITECTURE §Contracts — node format.)
- [ ] **Unit test:** round-trip — load a fixture `model.yaml` that carries a
      comment and custom spacing, `set_weight` on one node, reload; the changed
      weight is persisted and the comment + formatting are byte-preserved
      (ruamel), against a temp file (no paid/network call).
- [ ] Ties to the phase DoD ("`model.yaml` loads into 14 Node objects").

### SROTAS-003 — `core/events.py` — `events.sqlite` (WAL) append + read

**Description:**
The event journal seam. `core/events.py` creates `memory/events.sqlite` in WAL
mode with the minimal schema and provides append + read helpers. This pins the
**minimal events-schema contract** (§Contracts: `(id, ts, kind, node_id,
payload)`, `kind ∈ {bootstrap, feedback, weight_update}`).

**What needs to be done:**
- Create `events.sqlite` (WAL) with `events(id, ts, kind, node_id, payload)` —
  `payload` a JSON string; `kind` constrained to
  `{bootstrap, feedback, weight_update}`. Do not extend the schema.
- `append_event(kind, node_id, payload)` and `read_events(...)` (all / by kind).
- `events.sqlite` is runtime data — ensure it is gitignored (not tracked).

**Dependencies:** None

**Expected result:**
Events can be appended to `memory/events.sqlite` and read back in order.

**Acceptance criteria:**
- [ ] Fresh DB is created in WAL mode with exactly the 5-column schema.
- [ ] **Contract test:** the events schema is pinned to
      `(id, ts, kind, node_id, payload)` and `kind` is limited to the three
      allowed values — a fourth kind or an extra column is rejected.
      (ARCHITECTURE §Contracts — events schema.)
- [ ] **Unit test:** append two events (bootstrap + feedback-shaped payload),
      read them back and assert order, `kind`, `node_id`, and JSON payload
      round-trip, against a temp DB (no paid/network call).
- [ ] `events.sqlite` is confirmed gitignored.

### SROTAS-004 — Bootstrap provenance event on package creation

**Description:**
Wire the memory package's origin into the journal: creating/initializing the
package writes one `bootstrap` event recording the model's provenance (Lumi
facts, snapshot 2026-07-04). This is the join of SROTAS-002 (14 nodes) and
SROTAS-003 (events writer) and closes the phase DoD.

**What needs to be done:**
- An init path (a small `memory` init function / one-shot) that, on first
  creation, appends a single `bootstrap` event whose payload records: source
  (Lumi facts layer), snapshot date `2026-07-04`, and node count (14).
- Idempotent: initializing again does not append a duplicate bootstrap event.
- No scoring/embedding/collector code — this is memory-package wiring only
  (phase Out-of-scope).

**Dependencies:** SROTAS-002, SROTAS-003

**Expected result:**
After package creation, `events.sqlite` holds exactly one `bootstrap` event
tracing the 14-node model to the Lumi facts snapshot.

**Acceptance criteria:**
- [ ] Initializing the package writes exactly one `bootstrap` event with the
      provenance payload (source + snapshot date 2026-07-04 + count 14).
- [ ] Re-init does not duplicate the bootstrap event (idempotent).
- [ ] **Unit test:** init against a temp memory dir → `read_events(kind=
      "bootstrap")` returns one row with the expected payload; a second init
      leaves the count at one (no paid/network call).
- [ ] Ties to the phase DoD ("`events.sqlite` holds the bootstrap event").
