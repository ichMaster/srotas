# v0.3 — GitHub Issues

Issues for phase **v0.3 — Scoring + CLI preview** (version **v0 — Prototype**),
derived from the per-phase Tasks in [ROADMAP.md](../../ROADMAP.md) (§0.3) and the
contracts in [ARCHITECTURE.md](../../ARCHITECTURE.md) (§Scoring, §Item store,
§Configuration, §Contracts that must not drift). This file is scoped to a single
phase; IDs continue from the previous phase (SROTAS-008 →
**SROTAS-009…012**).

This phase makes items *rank* against the model, verified from the terminal
before any UI. It adds the Voyage embedding client (httpx, `input_type=document`
for items / `query` for centroids) with the DB embedding cache; the scoring pass
(`core/scoring.py`) that computes node centroids on the fly, gates on the
**pure cosine** (0.35 start, config), ranks by `cosine × weight`, and writes
`score`/`top_node`; a backfill for the `embedding` NULLs left by 0.2; and a CLI
preview to eyeball the top-20 and calibrate the threshold. No web, no LLM rerank.

## v0.3 — Scoring + CLI preview

### SROTAS-009 — Voyage embedding client (httpx) + DB embedding cache

**Description:**
The embedder seam. A Voyage client makes a direct HTTP POST via httpx to
`api.voyageai.com/v1/embeddings` (model **voyage-3**), with `input_type=document`
for items and `query` for centroids; the `VOYAGE_API_KEY` comes from config.
Item vectors are cached as BLOBs in `items.sqlite` so a re-score after a weight
change is free. This pins the **embedder seam** and the **cache-reuse** behavior;
scoring has **no LLM call** — the only network here is the embedding POST.

**What needs to be done:**
- `embed(texts, input_type)` over httpx → vectors (voyage-3); `input_type`
  is `document` for items and `query` for centroids.
- Add `VOYAGE_API_KEY` to `core/config.py` + `config.example.toml`.
- Cache item vectors in the `items.sqlite` `embedding` BLOB column; a cached
  item is **not** re-embedded on a subsequent pass.
- All Voyage HTTP is **mocked** in tests (a mock embedder returns fixed
  vectors) — never a paid/live call in CI.

**Dependencies:** SROTAS-005, SROTAS-006

**Expected result:**
Items can be embedded once via Voyage and reused from the DB cache thereafter.

**Acceptance criteria:**
- [ ] `input_type` is `document` for items and `query` for centroids (assert the
      request payload against a mock).
- [ ] **Contract test:** the embedder seam is pinned — items use
      `input_type=document`, centroids use `query`, model is `voyage-3`, key from
      config; no LLM call in the scoring path. (ARCHITECTURE §Scoring,
      §Contracts.)
- [ ] **Unit test:** the embedding cache is reused, not recomputed — embedding a
      batch twice calls the (mock) Voyage endpoint only for uncached rows;
      cached BLOBs are read back and decode to the original vectors (no
      paid/network call).
- [ ] `VOYAGE_API_KEY` is added to `config.example.toml`; `config.toml` stays
      gitignored.

### SROTAS-010 — `core/scoring.py` — centroids, cosine-only gate, `×weight` rank

**Description:**
The scoring stage. `core/scoring.py` computes each node's centroid as the mean of
its keyword vectors **on the fly** (no cache), embeds each item as
`title + ". " + summary`, applies the **relevance gate on the pure cosine**
(`cosine ≥ threshold`, 0.35 start in config), ranks eligible items by
`score = max over eligible nodes (cosine × weight)`, and writes `score` and the
argmax `top_node`. This pins the **cosine-only gate** contract — the threshold
never touches the weighted score.

**What needs to be done:**
- Centroid = mean of a node's keyword vectors (Voyage `query`), recomputed each
  pass; no centroid cache (MISSION §Scope, decision 1).
- Item text = `title + ". " + summary`; embed via SROTAS-009 (cached).
- Gate: an item is eligible for a node iff `cosine(item, centroid) ≥ threshold`
  (threshold from config, 0.35 start) — applied to the **pure cosine only**.
- Rank: `score = max over eligible nodes (cosine × weight)`; store `score` and
  the argmax node as `top_node`.
- Add the cutoff `threshold` to `core/config.py` + `config.example.toml`.

**Dependencies:** SROTAS-009, SROTAS-002

**Expected result:**
A scoring pass writes `score` and `top_node` for every item that clears the
cosine gate against at least one node.

**Acceptance criteria:**
- [ ] Scoring math is correct against a mock embedder (fixed vectors):
      centroids are keyword-vector means, `score = max(cosine × weight)`, and
      `top_node` is the argmax.
- [ ] **Contract test:** the relevance gate is **cosine-only** — a **low-weight**
      node with a **high cosine** still surfaces an item (eligibility uses pure
      cosine, never `cosine × weight`); an item below the cosine threshold for
      all nodes gets no `top_node`. (ARCHITECTURE §Scoring, §Contracts;
      MISSION decisions 21.)
- [ ] **Unit test:** the threshold is read from config and applied to the pure
      cosine; changing the weight reorders `score` but never changes
      eligibility, against mocks (no paid/network call).
- [ ] Ties to the phase DoD (items are ranked; `top_node` recorded).

### SROTAS-011 — Backfill `embedding` NULLs + collect→embed→score wiring

**Description:**
Bridge phase 0.2's NULL embeddings into the scored world: backfill the
`embedding` NULLs left by the Guardian collector, and establish the
collect → embed new → score pass so from here embedding is part of collection.
Reuses the cache from SROTAS-009 (only NULL rows are embedded).

**What needs to be done:**
- A backfill that embeds every item whose `embedding` is NULL (via SROTAS-009),
  then runs the SROTAS-010 scoring pass.
- Wire the ordered pass **collect → embed new items → score** (no scheduler yet —
  that is phase 0.4; this is the callable sequence).
- Idempotent: a second run embeds nothing new (cache) and re-scores from cached
  vectors.

**Dependencies:** SROTAS-009, SROTAS-010

**Expected result:**
Items collected in 0.2 get embeddings and scores; subsequent passes re-score for
free from the cache.

**Acceptance criteria:**
- [ ] Backfill embeds only rows with NULL `embedding`; already-embedded rows are
      untouched.
- [ ] **Unit test:** given items with mixed NULL/non-NULL embeddings, the
      backfill calls the (mock) embedder only for the NULL rows, then all items
      have `score`/`top_node`; a re-run makes zero embedder calls (no
      paid/network call).
- [ ] The collect → embed → score sequence is a single callable pass (no
      APScheduler — Out-of-scope this phase).
- [ ] Ties to the phase DoD (items are scored and re-scoring is free).

### SROTAS-012 — CLI preview (top-20) + threshold calibration

**Description:**
The terminal verification that closes the phase: a CLI preview prints the top-20
scored items (title, source, score, `top_node`) so the 0.35 cosine threshold can
be calibrated by eye before any UI is built.

**What needs to be done:**
- A CLI (e.g. `python -m core.scoring --preview`) that runs/loads the scoring
  pass and prints the top-20 items by descending `score`, each with its
  `top_node` and cosine.
- Support previewing at a given threshold so the value can be calibrated against
  real output; the calibrated 0.35 stays the config default.
- No web output (Out-of-scope this phase).

**Dependencies:** SROTAS-010, SROTAS-011

**Expected result:**
`--preview` prints a sensible ranked top-20 that lets the owner calibrate the
cosine threshold.

**Acceptance criteria:**
- [ ] The preview prints the top-20 items by descending score with
      `top_node`/cosine columns.
- [ ] **Unit test:** with a mock embedder and a small fixture item set, the
      preview output lists items in descending `score` order and honors the
      configured threshold (items below it are excluded), against mocks (no
      paid/network call).
- [ ] Ties to the phase DoD ("the CLI prints a sensible top-20; the 0.35 cosine
      threshold is calibrated against it; pytest green").
