# v0.2 — GitHub Issues

Issues for phase **v0.2 — Guardian collector + items.sqlite** (version
**v0 — Prototype**), derived from the per-phase Tasks in
[ROADMAP.md](../../ROADMAP.md) (§0.2) and the contracts in
[ARCHITECTURE.md](../../ARCHITECTURE.md) (§Item store, §Collectors,
§Configuration). This file is scoped to a single phase; IDs continue from the
previous phase (SROTAS-004 → **SROTAS-005…008**).

This phase makes real articles land on disk. It stands up the config seam
(`config.toml` + `core/config.py`, first key: Guardian), the item store
(`items.sqlite` with URL-primary-key dedup and a nullable `published_at`), and
the first collector (`collectors/guardian.py`, **one OR-joined request per
node** to respect the free tier), then wires them into a manual CLI run — no
scheduler, no embeddings yet (the `embedding`/`score`/`top_node` columns exist
but stay NULL for 0.3 to fill).

## v0.2 — Guardian collector + items.sqlite

### SROTAS-005 — `config.toml` + `core/config.py` (Guardian key)

**Description:**
The configuration seam. Introduce `config.toml` (gitignored) read via stdlib
`tomllib` through `core/config.py`, plus a committed `config.example.toml`
documenting the shape. This phase needs only the Guardian API key; the file
shape is introduced here and grows in later phases (Voyage/Anthropic keys,
threshold, interval, deltas, blacklist).

**What needs to be done:**
- `core/config.py` — read `config.toml` with `tomllib` (read-only; the file is
  hand-edited, no write path).
- Add the Guardian API key entry; expose it to the collector.
- Commit `config.example.toml` with the Guardian key placeholder (and a note
  that later keys arrive in later phases); ensure `config.toml` is gitignored.

**Dependencies:** None

**Expected result:**
`core/config.py` loads the Guardian key from `config.toml`, with a committed
example file documenting the shape.

**Acceptance criteria:**
- [ ] `config.toml` is gitignored; `config.example.toml` is committed and
      documents the Guardian key.
- [ ] `core/config.py` reads the Guardian key via `tomllib` (no write path).
- [ ] **Unit test:** load a fixture `config.toml` from a temp path → the
      Guardian key is returned; a missing key raises a clear error
      (no paid/network call).
- [ ] Ties to the phase DoD (the collector can obtain its API key from config).

### SROTAS-006 — `core/items.py` — `items.sqlite` (WAL), Item, URL-dedup upsert

**Description:**
The item store seam. `core/items.py` creates `items.sqlite` in WAL mode with the
full items schema and provides the `Item` class and an upsert keyed on URL. This
pins the **Item shape / items-schema contract** (§Item store). The full column
set is created now; this phase populates only the collected fields and leaves
`embedding`/`score`/`top_node` NULL for phase 0.3.

**What needs to be done:**
- Create `items.sqlite` (WAL) with
  `items(url PRIMARY KEY, source, title, summary, published_at, first_seen,
  embedding BLOB, score REAL, top_node TEXT)`.
- `Item` class covering those fields; `published_at` **nullable**.
- `upsert(item)` — insert or update keyed on `url`; a repeat URL does not create
  a second row (dedup by URL only, no simhash).
- `first_seen` set on first insert; leave `embedding`/`score`/`top_node` NULL.

**Dependencies:** None

**Expected result:**
Items can be upserted into `items.sqlite` and re-inserting the same URL updates
in place rather than duplicating.

**Acceptance criteria:**
- [ ] Fresh DB created in WAL mode with exactly the 9-column items schema.
- [ ] **Contract test:** the Item shape / items schema is pinned to
      `(url, source, title, summary, published_at, first_seen, embedding, score,
      top_node)` with `url` as PRIMARY KEY and `published_at` nullable.
      (ARCHITECTURE §Item store, §Contracts.)
- [ ] **Unit test:** upsert two items, then upsert one again with a changed
      title → row count stays 2 and the title is updated (URL dedup); a
      null-`published_at` item is accepted, against a temp DB (no paid/network
      call).
- [ ] Ties to the phase DoD (deduplicated items land in `items.sqlite`).

### SROTAS-007 — `collectors/guardian.py` — one OR-joined request per node

**Description:**
The first collector. `collectors/guardian.py` turns each node's `keywords` into
**one OR-joined Guardian search request per node** (not per keyword — free-tier
budget, ARCHITECTURE §Collectors) and normalizes the response into `Item` rows.
Reads nodes via `core/model.py` (SROTAS-002), the key via `core/config.py`
(SROTAS-005), and writes via `core/items.py` (SROTAS-006).

**What needs to be done:**
- Build one query per node: keywords joined with OR into a single
  `content.guardianapis.com/search?q=...` request (httpx).
- Normalize each result → `Item` (`source="guardian"`, title, summary from
  `trailText`, `url` from `webUrl`, `published_at` from `webPublicationDate`).
- Upsert via `core/items.py`; leave `embedding`/`score`/`top_node` NULL
  (embeddings are Out-of-scope this phase).
- All Guardian HTTP is **mocked** in tests — never a live/paid call in CI.

**Dependencies:** SROTAS-005, SROTAS-006, SROTAS-002

**Expected result:**
Running the Guardian collector over the 14 nodes fills `items.sqlite` with
normalized, URL-deduplicated Guardian items.

**Acceptance criteria:**
- [ ] Exactly **one** Guardian request is issued per node (OR-joined keywords),
      not one per keyword.
- [ ] **Unit test:** a mocked Guardian JSON response → normalized `Item`s with
      the expected field mapping (title/summary/url/published_at/source).
- [ ] **Unit test:** the request query for a node is built from that node's
      keywords joined with OR (assert the `q` parameter), against a mock — no
      live call.
- [ ] **Unit test:** two nodes returning the same article URL produce a single
      row (URL dedup carries through the collector).
- [ ] Ties to the phase DoD (Guardian items across the 14 nodes, deduplicated).

### SROTAS-008 — Manual CLI collect run across the 14 nodes

**Description:**
Wire the collector into a manual entry point (no scheduler — that is phase 0.4).
A CLI run loads the model, runs `collectors/guardian.py` across all 14 nodes,
and reports how many items were written/deduplicated. Closes the phase DoD.

**What needs to be done:**
- A small CLI (e.g. `python -m collectors.guardian` or a `collect` entry) that
  loads the 14 nodes, collects Guardian items for each, upserts them, and prints
  a summary (fetched / new / deduped counts).
- No APScheduler, no embedding pass (both Out-of-scope this phase).

**Dependencies:** SROTAS-007

**Expected result:**
One manual command fills `items.sqlite` with deduplicated Guardian items across
the 14 nodes and prints a run summary.

**Acceptance criteria:**
- [ ] The CLI runs a full collect over the 14 nodes and reports new/deduped
      counts.
- [ ] **Integration test:** with mocked Guardian responses for several nodes
      (including an overlapping URL across two nodes), the run writes the
      expected deduplicated Item set to a temp `items.sqlite` (no paid/network
      call).
- [ ] `embedding`/`score`/`top_node` remain NULL after the run (embeddings are
      phase 0.3).
- [ ] Ties to the phase DoD ("a collector run fills `items.sqlite` with
      deduplicated Guardian items across the 14 nodes; pytest green").
